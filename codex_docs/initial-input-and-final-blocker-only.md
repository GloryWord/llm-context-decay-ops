# 처음 요청 / 마지막 막힘만 보는 운영 가이드

이 문서는 **바쁜 사용자용 가장 짧은 안내문**입니다.
핵심은 두 가지뿐입니다.
- 새 작업을 시작할 때는 **지금 보고 있는 Codex 터미널 대화창에** 첫 요청을 보낸다.
- 끝까지 해결되지 않는 막힘이 생겼을 때만 Codex가 마지막 막힘 보고를 올린다.

전제는 **Codex + OMX가 주체**라는 것이며, maestro 전용 `MJ_Codex` 보조 구성은 여기 포함하지 않습니다.

---

## 1. 어디에 어떻게 입력하나
새 작업을 시작할 때는 **평소처럼 Codex 터미널에 메시지를 입력하면 됩니다.**

꼭 아래 형식을 한 글자도 안 틀리게 지킬 필요는 없습니다.
다만 처음 요청에 아래 항목이 들어 있으면, Codex가 덜 헷갈리고 더 빨리 움직일 수 있습니다.

## 2. 처음 요청할 때 쓰면 좋은 형식
아래 형식을 그대로 붙여 넣어도 되고, 자연스럽게 풀어서 써도 됩니다.

```text
[처음 요청]
목표:
범위 내:
범위 밖:
완료 기준:
참고 파일/문서:
```

### 각 항목 뜻
- `목표` = 무엇을 해달라는 것인지
- `범위 내` = 이번 작업에서 건드려도 되는 파일, 폴더, 주제
- `범위 밖` = 이번에는 건드리지 말아야 할 것
- `완료 기준` = 어디까지 되면 끝났다고 볼지
- `참고 파일/문서` = 먼저 읽어야 할 파일이나 문서

### 예시 1
```text
[처음 요청]
목표: judge.py와 run_experiment.py를 고쳐서 다시 실행 가능한 상태로 만들어라
범위 내: src/evaluation, scripts/run_experiment.py, tests
범위 밖: 보고서 문구 수정
완료 기준: 관련 테스트와 정적 검증이 통과하고, 영향 받은 문서가 있으면 같이 업데이트
참고 파일/문서: CODEX.md, docs/[Capstone]_이종웅_Lit_Review_and_Exp_Design_22110157.md
```

### 예시 2
형식을 딱 맞추지 않고 이렇게 보내도 됩니다.

```text
run_experiment.py와 judge.py만 이번에 손봐줘.
tests까지는 수정 가능하고, 보고서는 건드리지 마.
관련 테스트가 통과하면 끝으로 보자.
먼저 CODEX.md와 judge.py를 읽고 시작해줘.
```

## 3. 마지막 막힘 보고는 누가 쓰나
아래 양식은 **사용자가 미리 쓰는 양식이 아닙니다.**
Codex가 끝까지 스스로 해결하지 못했을 때, 마지막에 사용자에게 올리는 보고 형식입니다.

```text
[최종 막힘 보고]
문제:
이미 확인한 것:
아직 확인 못한 것:
선택지:
권장안:
```

### 예시
```text
[최종 막힘 보고]
문제: experiment_summary.json이 없어 보고서 숫자의 근거를 확정할 수 없음
이미 확인한 것:
- report 문구 위치 확인
- generate_report.py 집계 로직 확인
- figure 경로 확인
아직 확인 못한 것:
- 최신 machine-readable summary artifact
선택지:
A. generate_report 재실행
B. 관련 문구를 아직 확인되지 않음(UNVERIFIED)으로 낮춤
권장안: A
```

## 4. Codex가 내부적으로 처리하는 방식
사용자가 아래 이름들을 외울 필요는 없습니다.
다만 어떤 종류의 일을 시키면 Codex가 보통 어떻게 움직이는지 짧게 적어두면 이렇습니다.

| 작업 유형 | Codex가 주로 쓰는 방식 | 뜻 |
|---|---|---|
| 연구 설계 / 범위 조정 | `$deep-interview` 또는 `$ralplan` | 먼저 질문하거나 계획을 세워 범위를 굳힘 |
| 코드 / 파이프라인 수정 | solo 또는 `$team` | 혼자 처리하거나, 구현과 검증을 나눠서 처리 |
| 실험 실행 / 재실행 | `$ralph` | 오래 걸리는 작업을 계속 밀고 가며 중간 상태를 관리 |
| 보고서 / 발표 / 교수 피드백 문서 수정 | solo Codex, 필요 시 `gemini_only` | 먼저 숫자와 근거를 맞춘 뒤 문서를 수정 |

### 내부 공통 규칙
- 먼저 근거 자료부터 확인한다.
- 보통 한 쪽은 작업하고, 다른 한 쪽은 검증한다.
- Codex/OMX 모드에서는 `MJ_Codex`를 별도 외부 reviewer/helper로 두지 않는다.
- 검토자만 여러 명 늘리는 식으로는 운영하지 않는다.
- 근거 파일이 없으면 완료라고 하지 않고 `UNVERIFIED`로 둔다.
- 같은 막힘이 두 번 반복되면 사용자 판단이 필요한 상황으로 올린다.

## 5. 바로 써도 되는 시작 문구
짧게 지시하고 싶다면 아래처럼 보내면 됩니다.

### A. 설계성 작업
```text
plan this: Q4를 이번 학기 범위에서 제외하고 실험 설계를 다시 고정해줘
```

```text
deep interview: 실험 규모를 줄여야 하는데 무엇을 버릴지 먼저 정리해줘
```

### B. 코드 수정
```text
keep going: src/evaluation과 tests만 손봐서 scoring drift를 줄여줘
```

```text
team: run_experiment.py와 judge.py를 같이 손보되, 구현과 검증을 나눠서 진행해줘
```

### C. 실험 실행
```text
ralph: full cases 기준으로 summary와 figures를 다시 생성해줘. 막히면 마지막에만 알려줘
```

### D. 문서 / 발표 / 교수 피드백 반영
```text
이 피드백으로 final_report.md만 수정해. 숫자가 바뀌면 근거도 같이 확인하고, 마지막 막힘만 보여줘
```

---

## 6. 요약 운영 원칙
- 사용자는 시작할 때 첫 요청만 주면 된다.
- 중간 과정은 Codex가 맡는다.
- 정말 막혔을 때만 마지막 막힘 보고를 본다.
- 외부 reviewer는 기본값이 아니라 필요할 때만 붙는 보조 경로다.
- `MJ_Codex`는 이 Codex/OMX 운영 계약에서 별도 보조 경로가 아니다.
- 근거가 없으면 완료가 아니라 `UNVERIFIED`다.
